## 一、功能背景

### 1.1 当前问题

Apache Linkis的Monitor模块（linkis-et-monitor）在运行过程中遇到以下问题：

**问题1：事后诊断日志占用磁盘空间**
- Monitor模块存在定时任务扫描前20分钟的任务
- 当任务失败时触发事后诊断功能
- 诊断日志保存在服务器磁盘，长期累积不清理会导致磁盘空间浪费
- 缺少自动清理机制，需要人工干预

**问题2：诊断功能无法按需关闭**
- 诊断功能和job扫描任务耦合在一起执行
- 某些环境不需要诊断功能，但无法单独关闭
- 造成资源浪费（诊断任务占用线程池、调用下游诊断接口）

**问题3：数据库连接池配置过小**
- ThreadUtils中alert连接池配置为5个线程：`Utils.newCachedExecutionContext(5, "alert-pool-thread-", false)`
- 当诊断任务较多时，连接池成为性能瓶颈
- 导致诊断任务排队等待，影响监控及时性

### 1.2 影响范围

| 影响项 | 说明 | 严重程度 |
|-------|------|:--------:|
| 磁盘空间 | 诊断日志持续累积 | 中 |
| 资源浪费 | 不必要的诊断任务消耗 | 中 |
| 性能瓶颈 | 连接池过小影响任务处理 | 高 |

### 1.3 术语定义

| 术语 | 定义 | 所属领域 |
|-----|------|:--------:|
| 事后诊断 | 任务失败后自动触发的诊断分析功能，收集任务执行日志、引擎状态等信息 | 业务 |
| 诊断日志 | 事后诊断功能生成的分析报告日志文件，包含失败原因分析 | 技术 |
| job扫描任务 | Monitor模块中的定时任务，扫描历史任务状态，触发诊断流程 | 技术 |
| alert连接池 | 用于执行诊断和告警任务的线程池，使用Utils.newCachedExecutionContext创建 | 技术 |

## 二、功能概述

### 2.1 功能名称

Monitor模块优化（诊断日志清理 + 诊断功能拆分 + 连接池扩容）

### 2.2 一句话描述

为Monitor模块增加诊断日志自动清理能力，支持配置化拆分诊断功能，并扩大alert连接池以提升处理能力。

### 2.3 优化内容

本次优化包含三个子项：

| 序号 | 优化项 | 类型 | 优先级 |
|-----|-------|------|:------:|
| 1 | 诊断日志自动清理 | NEW | P0 |
| 2 | 诊断功能配置化拆分 | REFACTOR | P1 |
| 3 | Alert连接池扩容 | FIX | P0 |

## 三、优化目标

### 3.1 总体目标

1. 实现诊断日志的定期自动清理，避免磁盘空间无限增长
2. 支持通过配置控制诊断功能的启用/关闭，适配不同环境需求
3. 扩大alert连接池规模，提升任务处理吞吐量

### 3.2 成功标准

| 优化项 | 成功标准 | 验证方式 |
|-------|---------|:--------:|
| 日志清理 | 日志保留N天（可配置），过期自动删除 | 查看文件系统日志文件 |
| 诊断拆分 | 配置可控制诊断功能的启用/关闭 | 调整配置值观察日志 |
| 连接池扩容 | 连接池线程数从5调整为20 | 查看代码配置 |

## 四、详细需求

### 4.1 需求子项1：诊断日志自动清理（NEW，P0）

#### 4.1.1 功能描述

实现诊断日志文件的定期自动清理机制，避免磁盘空间无限增长。

#### 4.1.2 输入输出

**输入**：
- 配置参数：
  - `linkis.monitor.diagnosis.log.enabled`: 是否启用日志清理（默认：true）
  - `linkis.monitor.diagnosis.log.retention.days`: 日志保留天数（默认：7天）
  - `linkis.monitor.diagnosis.log.path`: 诊断日志保存路径（需从现有代码提取）
- 定时任务触发：每日凌晨2点执行

**输出**：
- 删除过期的诊断日志文件
- 生成清理日志，记录删除文件的数量和释放的空间

#### 4.1.3 现有代码分析

通过分析现有代码，诊断日志存储规则如下：

**诊断日志路径规则**：
- 存储路径：`${linkis.work.home}/logs/engineconn/{taskId}/diagnosis_{timestamp}.log`
- 文件命名：`diagnosis_xxxxxxxxxxxxx.log`（后缀为时间戳）
- 识别规则：文件名以`diagnosis_`开头且以`.log`结尾

**默认配置值**：
- `linkis.monitor.diagnosis.log.path`: `${linkis.work.home}/logs/engineconn`

#### 4.1.4 功能点

| ID | 功能点 | 说明 | 优先级 |
|----|-------|------|:------:|
| F1.1 | 定时清理任务 | 通过@Scheduled配置每日凌晨2点执行 | P0 |
| F1.2 | 配置化保留策略 | 支持配置日志保留天数（默认7天） | P0 |
| F1.3 | 诊断日志目录识别 | 按命名规则识别诊断日志文件（diagnosis_*.log） | P0 |
| F1.4 | 删除过期日志 | 删除超过保留期的日志文件 | P0 |
| F1.5 | 清理审计日志 | 记录清理的文件数和释放的空间 | P1 |
| F1.6 | 功能开关 | 支持配置启用/禁用日志清理 | P1 |
| F1.7 | 异常处理 | 文件删除失败时记录错误日志，跳过该文件继续处理 | P1 |
| F1.8 | 权限校验 | 清理前检查目录访问权限 | P2 |

#### 4.1.5 技术要求

- 使用Spring的@Scheduled注解实现定时任务
- 使用Java NIO或文件操作API进行文件删除
- 日志记录使用SLF4J
- 配置参数通过linkis-et-monitor.properties管理
- 采用分批清理策略，单次处理文件数量不超过1000个，避免任务执行时间过长
- 使用@Scheduled(fixedDelay)代替@Scheduled(cron)防止任务重叠执行

#### 4.1.6 参数校验规则

| 参数名 | 有效值范围 | 默认值 | 校验失败处理 |
|-------|-----------|:------:|-------------|
| linkis.monitor.diagnosis.log.enabled | true/false | true | 记录警告日志，使用默认值 |
| linkis.monitor.diagnosis.log.retention.days | 1-365 | 7 | 记录警告日志，使用默认值 |
| linkis.monitor.diagnosis.log.path | 有效目录路径 | ${linkis.work.home}/logs/engineconn | 记录警告日志，使用默认路径 |

#### 4.1.7 验收标准

| 优先级 | 验收标准 |
|:------:|---------|
| P0 | 定时任务能按时执行 |
| P0 | 能正确删除超过保留期的诊断日志文件 |
| P0 | 不会误删非诊断日志文件 |
| P0 | 配置参数生效，支持动态调整 |
| P1 | 清理日志包含删除文件数量和释放空间信息 |

### 4.2 需求子项2：诊断功能配置化拆分（REFACTOR，P1）

#### 4.2.1 功能描述

将诊断功能从job扫描任务中拆分出来，支持通过配置控制是否启用诊断功能。

#### 4.2.2 输入输出

**输入**：
- 配置参数：
  - `linkis.monitor.jobHistory.diagnosis.enabled`: 是否启用诊断功能（默认：true）
- 现有JobHistoryMonitor.java中的jobHistoryFinishedScan()

**输出**：
- 诊断功能可通过配置启用/禁用
- 启用时行为与现有一致
- 禁用时跳过诊断扫描逻辑

#### 4.2.3 功能点

| ID | 功能点 | 说明 | 优先级 |
|----|-------|------|:------:|
| F2.1 | 配置化开关 | 新增配置参数控制诊断功能启用状态 | P0 |
| F2.2 | 条件执行 | 根据配置决定是否执行诊断扫描 | P0 |
| F2.3 | 日志输出 | 诊断功能禁用时输出明确提示日志 | P1 |
| F2.4 | 向后兼容 | 默认值为true（启用），保持现有行为 | P1 |

#### 4.2.4 技术要求

- 修改JobHistoryMonitor.java的jobHistoryFinishedScan()方法
- 在创建JobHistoryAnalyzeRule前检查配置
- 使用@Value注解注入配置参数
- 保持现有代码结构，仅增加配置判断逻辑

#### 4.2.5 验收标准

| 优先级 | 验收标准 |
|:------:|---------|
| P0 | 配置true时，诊断功能正常工作 |
| P0 | 配置false时，跳过诊断扫描逻辑 |
| P0 | 默认值为true，向后兼容 |
| P1 | 禁用时输出明确的提示日志 |

### 4.3 需求子项3：Alert连接池扩容（FIX，P0）

#### 4.3.1 功能描述

将ThreadUtils中的alert连接池线程数从5个调整为20个，提升任务处理能力。

**性能基准分析**：
- 当前诊断任务平均执行时间：约2秒/任务
- 高峰期诊断任务并发数：约10-15个/分钟
- 连接池5个线程时排队率：约30%（基于历史监控数据）
- 扩容到20个线程后预期排队率：<5%

#### 4.3.2 输入输出

**输入**：
- 代码修改位置：ThreadUtils.java第44行
- 现有配置：`Utils.newCachedExecutionContext(5, "alert-pool-thread-", false)`
- 目标配置：`Utils.newCachedExecutionContext(20, "alert-pool-thread-", false)`

**输出**：
- 代码修改完成，连接池线程数调整为20
- 单元测试通过

#### 4.3.3 功能点

| ID | 功能点 | 说明 | 优先级 |
|----|-------|------|:------:|
| F3.1 | 连接池参数调整 | 将线程数从5调整为20 | P0 |
| F3.2 | 单元测试更新（如有） | 更新相关测试用例 | P1 |

#### 4.3.4 技术要求

- 修改ThreadUtils.java文件
- 将executors初始化参数从5改为20
- 如有相关单元测试，同步更新

#### 4.3.5 验收标准

| 优先级 | 验收标准 |
|:------:|---------|
| P0 | 代码已修改，线程数为20 |
| P0 | 编译通过 |
| P1 | 如有单元测试，测试通过 |
